Compiling and Building a Project

In order to allow verification and debugging of a design, you have to build an executable for the design and its test bench. Vista organizes your design environment to facilitate this process. Following this build process, the design is run.

The diagram below describes workflow involved in building an executable from your SystemC source files when working in Vista.

Figure 10‑1. Project Workflow

As can be seen above, when a project is compiled, an object file (*.o) is generated from each C++ source file. During the first or pre-processing phase of compilation, the contents of the included headers files are integrated into the design. This significantly increases the size of the object files.

The size of the object files is yet further increased by debugging-related information (functions, line numbers, etc.) written to these files by the g++ compiler. In fact, the g++ compiler used by Vista generates more detailed and fuller debugging information as a result of its modification by Mentor Graphics.

In order to minimize the impact of these two factors on the size of the resulting executable, at this stage only the executable statements derived from the source C++ file need be retained in the object files. To achieve this, Vista utilizes the objdump utility provided as part of gnu's binutils library to extract the debugging-related information. (Mentor Graphics modified this utility so that the extracted information is easier to read.)

The extracted information is first merged to eliminate unnecessary repetitions of the extracted header-related data, and then is written to a separate database that stores information related to the code derived from the debugging-related information.

Following extraction, the generated object (*.o) files contain only the pure logic of the design. It is these files that are then linked, the result of this process being the input for building the executable.
