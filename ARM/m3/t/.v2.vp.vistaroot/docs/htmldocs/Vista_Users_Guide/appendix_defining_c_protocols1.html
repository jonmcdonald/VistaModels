Using Protocols

Protocol and Transaction Basics

Vista's TLM models are based on pre-defined protocols and message types. All transaction-level data transfers are carried through ports and channels, where each transaction adheres to a specific protocol type. (Note that in HDL models, a protocol is usually implemented by multiple signals connected to a bus. The signals are abstracted into a single high-level port or channel and signal-level activity is abstracted to transactions and messages in the TLM model.)

Understanding how Vista views messages, transactions, and protocols is important for understanding how Vista and Vista Power work:

  • Messages - A Message applies to a single bus operation (usually conducted over one or a few clock cycles) representing a well-defined state of the bus and its related signals. A message is an atomic element that Vista handles: For example, BUS_REQ, BUS_GRANT, or READ_REQ.
  • Transactions - A Transaction is a well-defined sequence of one or more such messages that forms one specific bus transaction, such as READ, WRITE, UNDEF_READ, or UNDEF_WRITE. It executes complete read or write transactions, single or sequential (burst).
  • Protocols - A Protocol is the finite set of all defined Transactions on a bus, such as AMBA AHB (Advanced High-Performance Bus), PCI, or USB.

Vista can tune its engine and modeling granularity level to be protocol-aware or support a higher-level abstraction or a generic protocol. This allows modeling at the Protocol level or modeling Protocol-independent behavior.

Vista provides a set of pre-defined protocols, listed below. These can be viewed in a library in the Vista Browser. Users may also create their own custom protocols and compile them into Vista, via the parse_protocol_source command.

The Protocols are defined using SystemVerilog state-machine descriptions. Each state-machine monitors a hardware channel of the protocol.

Each protocol includes:

  • Declaration of protocol parameters where each corresponds to an HDL signal;
  • Declaration of protocol messages, and transaction-level operations such as Read and Write functions;
  • Declaration of the protocol state machine(s). The state machine defines all the valid paths from Idle to Idle state. (Via a menu option for a selected protocol, you can display a graphical representation of the state machine for the protocol.)

Once a protocol is created and compiled into Vista, it can be used just as any pre-defined standard protocol and assigned to any of the model's ports (see Interface under "Defining a Model").
