Extracting Power — Toggle Rates Based Method

A signal toggle rate is closely connected with the power consumed. If you know the capacitance and toggle rate of all the signals you know precisely the consumed power. Vista power approximates this by finding a (small) subset of the signals whose changing toggle rate through time represents the power behavior of the component. Vista Power toggle rates based learning method includes finding the most active signals of the component and learning the power, based on the toggle rates of a subset of the signals.

Power Learning Flow
  1. Create Vista RTL implemented model and setup input files (set of groups including VCD file + power dump file + DUT inside vcd file).
  1. Using Vista capabilities, extract most active signals from VCD input files. Based on the most active signals list, create a list of inputs signals to the learning engine.
  1. Run learning engine. As a result of learning procedure, get the decision tree and list of HDL signals, participating in learning. If the learning result is not acceptable, edit the list of the inputs signals and/or change the learning arguments and rerun the learning engine.
  1. Associate the HDL signals, participating in learning, with SystemC functional model ports.
  1. Generate code for the power model and add it to the project.
  1. Instantiate the power model in the design and bind its ports with the SystemC functional model ports.
Create Vista RTL Implemented Model

To create a Vista RTL implemented model, use the File > New RTL impl model menu item then set the input files in the Model dialog Extraction > RTL power inputs tab sheet (Figure 7‑37) or use the tcl function create_tlm_model with the following arguments: 

     -implementation-kind rtl 
     -rtl-power-inputs {{vcd file1 power dump file1 DUT1} {vcd file2 power dump
           file2 DUT2}….  }

Figure 7‑37. RTL Power Inputs

The model is listed in the Viewer Browser.

Extract Most Active Signals

To extract the most active signals from the vcd inputs of the model, use the tcl API get_most_active_signals function as follows or select the model in the Vista browser then right-click and choose Get most active signals (you will need to specify the signal top percent and if wires are to be included in the Most Active Signals dialog as shown in Fig2).

get_most_active_signals library model [-p top_percent] [ -w wires_included]

where:

top percent — percent of most active signals to use

wires included off — most active signals list includes regs only

Figure 7‑38. Most Active Signals Dialog Box

The following two files are created in the RTL Power sub-folder of the model:

  • toggle_counts.txt — read-only file which includes the toggle counts of all signals that have a non-zero toggle count in the vcd files (in accordance with the wires included argument)
  • allowed_signals.txt — includes list of signals whose activity is an input of the power learning engine. Vista produces the initial contents of the file, which includes the top percent of signals that appear in the toggle_counts.txt file. You can edit the allowed signals list prior to learning.
Learn Power HDL

The learning engine can be invoked using the learn_power_hdl tcl API function as follows or by selecting the model in the Vista browser then right-clicking and choosing Learn Power HDL (you need to specify whether to use clustering, the maximum number of clusters, and accuracy required as shown in Fig3.

learn_power_hdl library model [-clustering_mode no | maximum_clustersnumber, accuracy]

Figure 7‑39. Learn Power HDL Dialog Box

The following files are created:

  • power_report.html (read-only file) — contains the a learning report that includes the accuracy of learning and a decision tree that shows the dependence between the toggle rates of the most active signals and overall power values. If the accuracy is not acceptable, edit the inputs of learning (in particular, the allowed_signals.txt file) and/or change the clustering mode learning.

The Learning Report includes following sections:

  • Title and Date of generation
  • List of RTL inputs (vcd file with point to DUT and Power Dump file table)
  • Case Count — number of segments, where the toggle rates were computed and, together with power values, were provided to the learning engine
  • Model Description — includes following sub-sections:
  • List of power values cluster centers
  • Decision Tree

The decision tree analysis is similar to the state base power decision tree analysis (see “Decision-Tree Analysis”), but the inputs of the decision tree are the toggle rates and not signal values.

Example:

dt(5) 

              if "TgR<\state [1:0]>" min=0,max=100 [1797.000000] < 100 
                    [662.000000]
                  if "TgR<\state [1:0]>" min=0,max=50 [662.000000] < 50
                    [208.000000]
                 then "22.495284" [208.000000/34.000000] >= 50 [454.000000]
              then "31.076564" [1135.000000/482.000000]
  • Model Error — weighted average model error in mW
  • map_signals_2_sc.txt — includes mapping between hdl signals that appeared in vcd files and SystemC ports that will appear in the power model. Vista generates the default mapping (the SystemC port name is TgR__HDL port name). You can edit the file prior to SystemC code generation, but do not change the HDL signals that appear in the file.

Example:

If the activity of HDL signal state[1:0] is influent to power changes, the following line appears in the default map_signals_2_sc.txt file: 

           {TgR \state [1:0]} TgR__state__1_0__ uint64

The line indicates that the toggle rates of HDL signal state[1:0] will appear in the generated SystemC code as the TgR__state__1_0 SystemC port and the appropriate model port should be of type uint64.

Generate Code

To generate SystemC code, right-click and choose Generate Code or us the tcl API function:

save_model model-path -generate

The files model_name_power_model.cpp and model_name_power_model.h are created.

The Power Model is a SystemC SC_MODULE that has ports which appear in the map_signals_2_sc.txt file.

Using A Power Model in a Design

After code generation, you must add the power model to the project in addition to the functional code. For example, if the functional code is located in files dma_sc.h and dma_sc.cpp, and the power learned model is DMA, both files dma_sc.* and the DMA model should be added to the project.

The functional model should have SystemC ports (not just internal design variables) that behave the same as the HDL ports and signals that participate in learning. The power model ports should be bound with such ports. That is, in the module that instantiated the functional code, you must add instantiation of the power model and for each power model port, binding between the functional model port and power model port. It enables viewing of the power graph in the analysis window during the simulation.

Example of Usage of Power Model in a Design

Sat that in the vcd file, the most active HDL signal is named “state[1:0]”.

File map_signals_2_sc.txt includes the following mapping:

  • Functional model dma_sc.h include port that behaves similar to HDL signal state[1:0]:
           sc_inout < unsigned long long > State;
  • Generated power model includes appropriate port:
           
           class DMA_power_model: public sc_core::sc_module
           {
           public:
           ……..
             mb::rtl_power::toggle_attribute< unsigned long long > TgR__state__1_0__;
           ….
           };

Say the dma functional model is instantiated in the SC_MODULE(top) (file top_sc.h), so inside SC_MODULE(top) you have the following code: 

     #pragma once 
      
     #include "systemc.h"
     #include "dma_sc.h"
     
     …
     SC_MODULE (top) {
     …….
       dma    d;
     …..
       SC_CTOR(top) : … d("Dma")
         {
     …
       }
     };

You need to add the power model instantiation there: 

     #pragma once
     
     #include "systemc.h"
     #include "dma_sc.h"
     
     #include "DMA_power_model.h"
     
     
     
     SC_MODULE (top) {
     ….
       dma    d;
       DMA_power_model d_power;
     ….
       SC_CTOR(top) : … d("Dma"),d_power ( "dma_power" )….
         {
     ….
         d.State ( d_power.TgR__state__1_0__ ); // bind functional model port with the appropriate power model port
     ….
       }
     };
Setting Sampling Interval

The sampling interval is the minimal time interval used to compute the toggle rate. The default sampling interval is 1 US. To set the sampling interval, use the simulation parameter power_sampling_interval, which is defined in the parameters file as follows: 

     path to design top.power model instantiation name.power_sampling_interval 
        = time interval value

For example, 

     top.dma_power.power_sampling_interval = 20 ns
Simulation of Power Model Outside of Vista

The same as for a state based non-TLM power model, you can create a power model inside Vista and simulate it outside of Vista. In addition to the requirements listed in “Using A Power Model in a Design”, you need to declare callback functions and update the makefile for the project:

The following callback functions should be declared: 

     Void _power_model_user_evaluate ( double power, sc_core::sc_time time) 
     Sc_core::sc_time get_power_sampling_interval ()

The following options should be added to the makefile:

On compilation of gcc:

           -DVISTA_POWER_EXT_MODEL -I vista_istallation/papoulis/include 
              -I vista_istallation/papoulis/include/mb_include

On link:

     vista_istallation/papoulis/arch/libst.esl.library.so  
     vista_istallation/papoulis/arch/vista-gcc-sc-version/esl.sc_sim.a
     vista-gcc-sc-version - ex., vista_gcc34_sc22

Where:

arch is linux32 or linux64
