Parameters

This optional task involves specifying parameters to be used for internal model processing. Parameters are set statically before simulation and can be accessed from within the model. The parameter values can also be set in the Parameter file.

The Parameter tab allows you to specify global declarations and to configure each instance of each of the models with specific parameters. These parameters are set statically for each simulation session and include global parameters such as clock, and local parameters such as operation mode or bus-configuration parameters.

Parameters can be set in a parameter text file. All parameters of all instantiated models in a design are grouped into a single unified list and can be assigned desired values.

A line that defines a parameter value takes the form: 

     instance_path.parameter_name = value

instance_path is the full SystemC path of a component instance in the simulation, with the top-level design, with each level of the hierarchy separated by the “.” character.

Note that a line which begins with the `#' char is treated as a comment.

If the environment variable VISTA_PARAMETERS_FILE is set, then its value is used as the name of the parameters file. If VISTA_PARAMETERS_FILE is not set, but SD_PARAMETERS_FILE is set, then the value of the SD_PARAMETERS_FILE environment variable is used. If neither are set, the default file name parameters.txt is used. The file should be in the current working directory.

For more information regarding specifying the parameterization file, see the System Architect tab in "Invoking Simulation".

Auto-Generated General Parameters

The auto-generated parameters of the model are:

  • clock - defines the general clock's value for the model. The default is `1 ns'.
  • verbose parameters - sets the verbosity of simulation printout for every model. Its default value is `True'.
  • protocol_name_clock - defines the default clock parameter for every protocol used in the model. Its default value is that of the general `clock' auto generated parameter.
  • port_name_pipeline_length - for every slave port defined in the model, this parameter indicates the number of threads created for activities that are initiated on transactions on that port. Its default value is 2.

When this parameter is set to `n', the slave callback of the model can be called n times in parallel. This means that before the callback finishes, it could be called for another scene.

  • mb_debug - The mb_debug parameter is used for automatic printing of debug information. When it is set to 1 then for every target callback (associated with target ports, registers and memory ranges) the following debug information is printed out: callback name, address and the corresponding data for write transactions.
  • nominal_voltage - The value of nominal_voltage represents the nominal voltage of the component and it is measured in Volts.

The default value is 1 Volt. Refer to the set_voltage function in Specifying Power Policies for further explanation on how it is used.


  Note

A warning is displayed if more than n callbacks are active in parallel.
 

Refer to Timing-dependent behavior in the Vista Modeling Guide for an explanation on the creation of threads in slave callbacks.

The following parameters are generated for every port that instantiates a protocol that is implemented with multiple state machines; for example, AXI:

  • port_name_outstanding - indicates the maximum number of transactions that can be issued in parallel on a master port. It also indicates the maximum number of transactions that can be processed in parallel by a slave port. Its default value is 1.
  • port_name_read_data_queue_size and port_name_write_data_queue_size - defines the read/write data queue size for master ports in bytes. Default value is `0'.

The size of the read/write transactions that are issued in parallel cannot exceed the queue size. If a transaction is issued when there is no more room left in the queue, the transaction is halted until one or more transactions are finished and space is freed in the queue. If the read data queue size is equal to zero, the size is unlimited.


  Note

For multi-channel buses; for example, AXI, the port_name_read_data_queue_size and port_name_write_data_queue_size parameters are not generated.

  • warning_level (string) - defines how warnings issued by the model are treated.

Possible values:

  • IGNORE: No messages
  • WARNING: Warning (SC_REPORT_WARNING) is issued
  • ONCE: Warning (SC_REPORT_WARNING) is issued only once for each kind of message
  • ERROR: Error (SC_REPORT_ERROR) is issued; Simulation is stopped
  • FATAL: Fatal (SC_REPORT_FATAL) is issued; Simulation is aborted (with stack/core)

This parameter can also be specified for model sub-components (registers, ports, memories, etc). Example:

           inst_path.Model.PV.port0.warning_level = ONCE
           top.bus.warning_level = IGNORE
  • dmi_enabled (bool) - defines whether model issues DMI requests.

This parameter can also be specified for model sub-component (registers, ports, memories, etc) and can be used also to view transactions (waves, tracers) while running in LT mode.

Examples:

           inst_path.Model.PV.port0.dmi_enabled = false
  • disable_dmi — In peripheral models, disables the register/memory ranges DMI
  • disable_dmi_memory, disable_fast_memory — In memory components, disables DMI returned from the components.
Specifying Parameters in the Modeling Dialog

To specify parameters for the model:

  1. In the Vista browser, double-click on the model.
  1. In Vista's modeling dialog, select the PV tab.
  1. Select the Parameters > User-defined tab:

Figure 7‑13. Vista's Modeling - User-defined Parameters Tab

  1. Enter the interface parameters in the table:
  • Enter the parameter's name in the Name field.
  • Enter the parameter's default value in the Default Value field and select its type in the Type field.
  1. Select the Parameters > Auto-defined tab to review the protocol-dependent parameters specified automatically by Vista.
  1. Modify the Default Value assigned to a parameter where necessary.

Figure 7‑14. Vista's Modeling - Auto-defined Parameters Tab

  1. Click Save.

The changes to the Model are saved.
