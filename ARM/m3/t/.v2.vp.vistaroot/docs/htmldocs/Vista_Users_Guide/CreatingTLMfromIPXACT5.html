Generated Models

Together with the new TLM model representing the configured ARM component (currently, NIC301), several other models are generated in the same output library. These additional models are used to create subcomponents of the main TLM model and serve to adapt port protocols, and to host the memory maps (for example, the register and memory range memory maps) for the main TLM model. The models of the subcomponents are created in the target library under a sub-directory with the same name as the main TLM model.

For each slave port, a new Interconnect model named protocol2AXIbitwidth is generated (for example: AHB2AXI32 or AHB2AXI64). One model is generated for all slave ports having the same protocol and same data width.

For each master port, a new Interconnect model named AXI2protocolbitwidth is generated (for example: AXI2APB32 or AXI2AHB64). One model is generated for all master ports having the same protocol and same data width.

A Bus model named axiBUS is generated. The Bus model has the same number of slave and master ports as the original ARM component, but all ports use the AXI protocol. The model encapsulates the set of switch blocks that implements the interconnections between input (slave) and output (master) ports. A layers parameter specifying all the connected ports is specified for the model.


  Note

Clock ports and ports named port_name_do_not_connect are not created for the TLM2.0 model.

A GPV model named register_block_name is generated. The GPV model contains all the needed registers defined in the ARM component.

An example of topology of a generated NIC301 TLM model is depicted in Figure D‑2. The top level NIC301 TLM model defines the following information:

  • external ports
  • master_n_base_address and master_n_size parameters for each master port; holds the memory map information
  • Layers parameter; holds slave to master connectivity information
  • Pipeline latency for each connected slave to master path
  • Declarations of all needed registers

The generation phase creates the internal hierarchy, assigning the needed information to the corresponding internal model (for example: layers and *_base_address/*_size parameters to the BUS model, registers to the GPV model, and pipeline latency data to the BUS model).
