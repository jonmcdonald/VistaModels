The Types

A set of new types is created to implement and make use of the new capability.

sc_port_vector

sc_export_vector

initiator_socket_vector

target_socket_vector

These types are defined respectively in:

vista_schematics/port_vector.h

vista_schematics/export_vector.h

vista_schematics/socket_vector.h

Being part of some MODULE, each of these types represents a vector of ports/exports/tlm_sockets/channels, and has the same graphical representation as an appropriated single object. For example, `initiator_socket_vector' is recognized as `tlm_initiator_socket'. It is represented by one PIN which can be bound to another PIN in the SCHEMATICS. The binding operation produces the `$bind' section in the generated code of the “top” module:

(in our examples “initiator” and “target” are modules instance name.) 

     $bind("initiator->initiator_socket","target->target_socket");
     initiator->initiator_socket.bind(target->target_socket);
     $end;

By default, method `bind()' assumes that `target->target_socket' is another `vector' of connectors (same as 'initiator->initiator_socket').

The default behavior may be overloaded by one of the following ways:

  1. Modifying the generated $bind section in the parent module. For example the line
           initiator->initiator_socket.bind(target->target_socket);

may be replaced by 

           multi_bind(initiator->initiator_socket, target->target_socket);

Note that the SCHEMATICS generation mechanism does not re-generate the code between `$bind' and `$end' markers, keeping the user's code as long as the “binding” LINE appears in the SCHEMATICS.

  1. Creating a sub-class from the initiator_socket/port/_vector and re-defining the `bind()' function in the derived class.

Please refer to: SocketVectorPrj, PortVectorPrj examples that demonstrate the usage of these classes in the design.
